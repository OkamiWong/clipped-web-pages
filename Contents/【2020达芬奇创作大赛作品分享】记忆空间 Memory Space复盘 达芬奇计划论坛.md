**记忆空间 Memory Space**

这是一个关于【复制空间】为主题的游戏。主角可以复制自己当前所在的房间，并且把它放置在合适的位置来帮助自己离开房间，通过关卡。每个房间都有需要收集的记忆——

\* 玩家从房间中醒来，却被告之身处于自己的 “记忆空“间” 中，只有探索空间才能恢复记忆。

随着记忆一点点被找回，玩家逐渐理解了自己的身份——一个特殊身份的警察：潜入他人记忆中的警察。

利用特殊研发的 “机器”，警察可以将自己的潜意识投入到罪犯的脑中，并且在对方记忆中直接与罪犯的潜意识沟通，引导对方暴露内心深藏的记忆，得到有用的信息。

拥有这样特殊警察身份的玩家，是被什么人潜入了记忆，在这个 “神秘人” 的引导下探索自己的记忆呢？

记忆中频频出现的女孩是谁？我为什么要去救她？

随着探索的深入与记忆恢复，我能想起来自己的真实身份吗？

关于我的故事，究竟是什么样的呢？\*

**做到的事**

一个【流程】完整的游戏。这是一个机制上所有功能实现完成，美感上各种装饰美化装配完毕，关卡上有起承转合，剧情上有开头结尾的完整故事的游戏。

我们团队做好的事情就是在我们的时间和能力能做到的范围内，完成了一个流程相对完整的游戏 demo。只游玩这个 demo，玩家也可以获得最核心的游戏体验。这个核心的体验是：

-   机制：游戏的最核心机制（复制房间）以及游戏整体框架的实现
-   关卡：基于**最核心机制**设计的有完整学习设计的关卡组合
-   美术：符合故事氛围的装饰与音乐音效
-   剧情：符合主题的有结构的完整故事，并且和已有的关卡和美术交叉结合

没有做到的事

一个【残缺】的游戏。这是一个因为开发时间不够为了优先完成度而抛弃了重要的设计，因为美术风格和核心玩法之间的冲突以及不足的完成度而改变画风，因为功能实现不足而只实现了最核心的部分舍弃部分美化的游戏。

我们团队没有做到的事，就是利用资源去扩张游戏的**表现**，而是只保证了核心的完成。

如果日后仍有机会，我们确实可以直接在已有 demo 的基础上扩张游戏的内容，但是这次比赛的时间已经结束了——

**创意初期：主题，脑暴，创意文档，游戏原型与 ONE PAGE DESIGN**

本次的主题是复制。

第一次团队会议的目标就是关于主题的脑暴。在脑暴之前以及讨论的时候，大家都一致觉得今次比较适合做解密类的游戏，所以脑暴和之后讨论的方向都是以解密游戏为前提的。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/6fa388fe-b1aa-48df-a164-90f6e9d68ad2.png)
](https://davinci-cdn.lilithgame.com/forum/fe445e8d-54d3-4d90-b897-60e3a7eb686e.png)

脑暴当然不可能最开始就能这么有逻辑——虽然每个人 idea 的产出都是基于自己思考的逻辑产生的。这是基于初始脑暴的产出按照和主题关联的逻辑整理好的思维脑暴图。我们基于每个分支都讨论了各自关于主题的想法，最后以脑暴阶段有最详细思路的几个 idea 为蓝本，对他们进行一个制作评估：

-   有趣：是否足够有创意
-   清晰：游戏目标和流程是否明确
-   完成度：预估的工作量是否可以达到很好的完成度
-   实现难度：是否有技术瓶颈

由于团队规模限制，我们只试着针对其中几个 idea 进行了制作评估，最后的 idea 定在了【复制空间】和【复制操作】上。

接下来粗略的评估已经不够用，我们便开始了创意文档的整理与游戏原型的制作，并在第二次团队会议（上展示：做好创意文档和游戏原型后邀请整个团队一起来验证，最后决定一个最后的方案。（以下是以【复制空间】为例）

-   创意文档其实只是一个让我们更能看清我们需要做什么的一个工具（其实是个思维导图，我自己习惯叫文档是因为平时写成文档的样子），在不考虑任何商业问题的情况下，我们从游戏的核心玩法 / 游戏循环等角度去总结我们要做一个什么样的游戏。  
    复制空间的主题就是以复制玩家所在的房间为核心玩法，通过房间内共享机关的状态以及连接房间之间的空间来完成一系列的解密。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/bfb52f73-567e-487c-b261-ecf84c42daca.png)
](https://davinci-cdn.lilithgame.com/forum/9574a2e5-f979-4b30-88bc-e1ce08e7ba91.png)

-   游戏原型其实只要是能玩的用什么做都行，最开始我家网还没搞定的时候我就直接拿纸糊了几个关卡，然后和队友视频通话展示我做的关卡的时候，以类似跑团的形式来让他们体验这个游戏是怎么玩的。
-   举个例子：（以 Tutorial1 为例）

    -   我：你们进入了房间内部。面前有一个黄色的开关和一个红色的看似可以穿过的闸口，在对面有一个绿色的出口，但似乎是关着的。
    -   队友：我选择打开开关。
    -   我：你走到开关附近，触发了开关。你发现绿色的出口打开了。
    -   队友：我选择穿过红色的闸口，从出口离开房间。
    -   我：你穿过了共色的闸口，但这时出口关闭了。与此同时你之前打开的开关被重置回关闭的状态。
    -   队友：（理解了碰到红色的闸口会重置开关）
    -   我：你的核心能力是：在房间内选择一个方向，并把当前的房间在这个方向上复制一份……
    -   队友：复制后的房间的开关可以影响我当前房间的开关吗？
    -   我：我不知道，但你可以试试……

    跑团玩法的核心就是模拟玩家在游玩关卡的状态：由于测试原型者对什么都一无所知，只能以尝试和玩耍的心态去与你设计好的原型关卡互动。你并不会提前告诉测试者核心道具的规则和通关的方法，而是只告诉他们作为一个玩家可以做的事情，让测试者以一个玩家的视角去体验你的原型关卡。  
    其实从这个阶段，基本的游戏玩法就已经可以确定了：玩家在一个房间内探索，并且与房间内的道具互动，或者使用【复制房间】的能力，以达到离开这个房间的目的。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/90e31e0c-82ed-4ddb-b210-15419957c090.png)
](https://davinci-cdn.lilithgame.com/forum/20eeeebe-7219-45b3-84e5-dee515bb33fb.png)

在验证了这个玩法并得到一致通过后，听到了 8/5 要交策划案的声音，就顺便做了一个 ONE PAGE DESIGN（8/5）。核心的想法是用一页纸的图文精简的表达你要做的游戏的核心内容，让阅读者只通过读图和少量的文字也能明白这个游戏是怎么玩的。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/cb8d13cd-0deb-4c0f-afbd-2b1b0f0bdad1.png)
](https://davinci-cdn.lilithgame.com/forum/6add7ddb-db09-444f-92a8-3d970dcf93fb.png)

**系统设计**

原型阶段的时候实际上就已经确定了玩法上的系统设计：游戏将以离开房间为目标，以各种机关和核心能力（复制房间）为手段。这个设计也并不是一拍脑袋就能产出的，实际上也考虑到了很多问题。

首先是核心道具的选择。设计的时候我考虑了以下几个点：

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/90f2ec2e-b293-491a-a354-91fb6a7af798.png)
](https://davinci-cdn.lilithgame.com/forum/1739376d-143d-4de8-9fdc-d959d868ac48.png)

1.  简单：最核心的机制要简单易懂，教学的时候很容易能弄明白。尤其是 3D 空间解密本来就对晕 3D 的人很不友好了，如果教学都搞不懂后面就很难办。
2.  创造矛盾：解密游戏的核心就是创造矛盾，给玩家创造一个看似两难的困境，再通过玩家的探索和学习去明白这个核心道具的新用法 / 新的理解角度
3.  多用途：同一个核心道具可以实现不同的功能。
4.  泛用性：核心道具要能和后续可能的其他解密道具结合起来，不能相互之间孤立，最好是前一个阶段对核心道具的学习能应用到新的可能的道具上。
5.  最好不要和已有的解密游戏有类似的道具。（我知道【复制空间】的机制很像传送门，因此会特意避免按钮 + 方块的解密组合。）

因此，核心道具就在备选的几个方案中选择了 开关 + 触发门（就是之前原型中的闸口）。在实际中的测试也表现很好，队友们都很喜欢这个设计。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/040ac7ce-934b-4211-a260-f7a7919c4325.png)
](https://davinci-cdn.lilithgame.com/forum/2e059b4b-853d-4958-bbe1-41ec18e37968.png)

接下来是核心能力。【复制房间】已经是确定了的，但是具体的规则应该怎么考虑呢？由于这是一个很强大的能力，泛用性也非常强，所以我觉得这里做减法会比较好，即限制玩家使用能力的方式和机会，一方面简化了关卡设计的难度，另一方面更易于玩家理解规则和应用能力。对于复制房间这个能力来说，限制复制的位置就是一个简单而优雅的解决方案。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/2731adcc-b90e-4bb3-99bc-681cf57c1c62.png)
](https://davinci-cdn.lilithgame.com/forum/e577023e-5e8f-438e-858f-54ea79263361.png)

核心能力的变形也很容易确认，原型阶段中考虑过旋转房间，实际上水平旋转 / 桶滚旋转 / 镜像复制 / 中心点对称复制都是可以选择的变形，并且每一个变形都可以引申出新的学习内容。考虑到我们的制作进度，我们计划以水平旋转 / 桶滚旋转作为可能的能力变形。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/d78e1789-a9cc-4740-940c-8308081e28a9.png)
](https://davinci-cdn.lilithgame.com/forum/7eaa63f8-037c-43b3-bda3-35b65f217b00.png)

最后最难决定的是次要道具。我的设想是设计一个类似开关 + 触发门的道具组，同时更多体现道具的多用途性比较合适。但是一个我认为更重要的设计标准是脱离核心能力于核心道具只会改变玩家空间位置的桎梏：如果我的核心游戏部分介绍了我的能力和互动只能改变我的位置，那么新的道具就要给玩家介绍改变空间位置以外的机制，这样才能保证游戏足够有趣和耐玩。

因此，我想更多体现出【复制】这一主题的存在，便在几个备选方案里了钥匙 + 钥匙孔的次要道具设计。玩家可以通过复制房间来将房间内的核心道具复制一份，并且可以在房间之间内自由运输被复制的道具——钥匙。在这样的设计中，一个房间内可能钥匙孔的数量会大于钥匙的数量，玩家需要通过复制房间并把钥匙运送回同一个房间内才能打开目标门（出口）。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/d201303d-d02f-4294-8e9d-c00abda20462.png)
](https://davinci-cdn.lilithgame.com/forum/8993cd87-476a-42a4-891a-ae4a1e64fcd1.png)

但是——这并不是一个好的设计。

首先最明显的，从泛用性考虑，这个机制和前一个核心道具（开关 + 触发门）是脱离开的，在玩家接受了开关 = 开启目标门（房间出口）的逻辑后，我需要重新设置教学关卡去重申新一组道具（钥匙 + 钥匙孔），这对关卡设计和整体玩家的心流设计是极其不友好的。

另一方面，关于钥匙孔的规则设计并不简单，甚至到后期有些过于复杂：

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/65795dc0-f752-4035-a9d5-3a4dfa6e9869.png)
](https://davinci-cdn.lilithgame.com/forum/2b307edf-1f48-45ca-9dcd-6a9fcbfe1316.png)

这一长串关于规则的描述其实只体验了钥匙 + 钥匙孔的三个特殊性：可以随着房间的复制而复制 + 钥匙孔（文档里被叫做钥匙台，因为想增加多用途方向的设计）高一格（游戏中房间像魔方一样由 3 _ 3 _ 3 的格子组成）并可以当垫脚石 + 钥匙台可以被推动。这些设计本来都是为了多用途考虑的，但是由于钥匙和钥匙孔组合的复杂性，导致它的多用途设计也没有做好。

因此，我重新设计了关于钥匙 + 钥匙孔，使得它符合我之前提到的设计原则：

_复制台_

-   可以随着房间的复制而复制。
-   可以一直触发开关。
-   可以被推动并且高一格。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/55341867-1ce1-40cc-81ac-bd69d6ebe444.png)
](https://davinci-cdn.lilithgame.com/forum/cec47110-bc47-440f-bc1a-7c6d172e8c90.png)

我保留了钥匙与钥匙孔的多用途设计，但把钥匙+钥匙孔简化成一个可以被推动的台子。而原来的 “把所有钥匙放在钥匙孔内才能打开房间” 直接取消，取而代之的是复制台可以“压住” 开关 + 触发门中的开关，使之保持触发状态。这样一来，复制台这个核心道具就能和开关与触发门紧密连接在一起。而多用途的设计（可推动 + 高一格）也能体现的更好，携带钥匙与房间之间也变成了可以将复制台在不同房间之间运输。

复制台的设计解决了钥匙 + 钥匙台过于复杂的设计（简单性），同时与之前的核心道具更紧密结合（泛用性），又考虑到了多用途的设计，是一个很适合在核心道具之后出场参与解密的次要核心道具。

最后是关于游戏的流程。最开始设想的流程是玩家可以在不同房间之间穿梭，在破解了一格房间的空间谜题后可以经过一段过渡的走廊 / 阶梯后到达下一个解密房间，在过渡的走廊 / 阶梯中可以缓解解密的氛围并引入更多剧情的介绍。尽管这个时候剧情仍然没有确定下来，之后可以直接填入过渡的空间中和解密空间中。考虑到制作流程很短就没有想更多了。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/af9670d6-9601-4275-9df5-693e897ac3e2.png)
](https://davinci-cdn.lilithgame.com/forum/aa76a302-5e36-4ed0-8809-26be39b19b83.png)

**关卡设计**

确定了基本机制后就是关卡了。按照之前房间之间相连的设计，关卡之间时线性连接的。因此只要考虑一个线性的学习流程。

所有的关卡设计都是对已有元素的拆解和重新教学，我自己的设计方法是把元素拆解后归类，并按照一定顺序教学给玩家。

比如，对于简单的【复制房间】，其实际上牵扯的逻辑如下（和程序协商调整后）：

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/861fc7f3-c2f1-4472-a6f8-abb118159a25.png)
](https://davinci-cdn.lilithgame.com/forum/df308a71-66a8-42e0-9b57-60edd246bdcb.png)

从中我们可以拆解的最基本的元素，也就是教学可以直接表现的规则元素有：

【可以选择不同的墙面复制房间，并只能开一个门（洞）】

【玩家可以通过这个门到达新的房间】

【可以在复制房间前选择复制房间的方向】

这些是基础教学可以直接告诉玩家，让玩家在自己的实践中学会的内容。玩家在知道了规则后会反复尝试并应用已经学到的东西，在不断的尝试中有一些新的实践元素可以拆解出来，比如：

【旋转房间可以不止一次，玩家可以旋转房间 180 度并复制】

【某些道具会堵住玩家的去路，玩家并不能越过这些障碍】

【某些道具可以把玩家引向更高的高度（斜坡）】

【复制的房间会一直保留】

【如果复制方向已经有房间，则不会复制新的房间而是只开门】

【按照玩家的移动速度，无法直接穿过空一格的平台】

【复制可以让我上高台】

在熟悉了操作之后，玩家拥有了可以理解复制仍然有一些新的元素，是玩家在解密过程逐渐发现，理解并利用的隐式元素。这一般也是让解密有趣的点，也是玩家一开始想不到的点，即玩家遇到矛盾的所在：

【虽然触发门会重置开关，但有时候为了打开所有开关可以先把自己困在触发门内】

【玩家虽然不能跳跃，但可以从高处落到低处完成位置移动】

【虽然有障碍会阻挡，可以利用已复制的房间绕过房间内的障碍】

【复制时需要考虑已复制房间的位置和安排，必要时需要考虑复制方向以便后续使用】

这三种可拆解的元素（规则元素，实践元素，隐式元素）对于玩家来说，都是可学习的信息，并且越靠后的元素学习起来就越难。这就是控制关卡难度的方式：同一关卡内需要学的元素越多越靠后，关卡就会越难。

在有了元素后，还要考虑如何把元素教给玩家：

-   规则元素一般是直接在教学关卡内直接学会的，【教学规则元素的关卡】一定要足够简单，尽量让玩家不产生有歧义的理解。
-   实践元素是需要反复练习的，因此有时候并不能只出现一次，而是需要在不同关卡内均出现，方便玩家通过操作去理解并熟悉规则。
-   隐式元素一般是作为解密的关键出现，是基于对规则的理解而产生的，所以一般与谜题的目标相关。重复隐式元素同样也是一种有效的教学手段。
-   关卡之间要考虑到难度的协调，可以参考心流曲线的使用。一个方案是，你可以试着把之前提到过的元素量化，然后计算关卡的难度（数字上），以此为目标去设计关卡。

按照这种方式设计完关卡后，还需要测试难度和微调各种元素的参与，因为不同的拆解方式会产生不同的元素，最终还是要以玩家游玩的体验为参考才能保证关卡有趣并难度适宜。最后大概考虑的是这样感觉的关卡设计。

我实际上是用 MagicaVoxel 做的关卡模型，这样也方便带入自己的视角去实际解密玩，不过调整难度并没有人帮我测试，所以实际上的难度控制和学习控制可能并没有那么理想（主要还是去掉了复制台这个道具导致只能尽可能用复制房间 + 旋转 + 开关 + 触发门拆出尽可能多的东西）

**团队**

和同校的两位一起组队，三个人正好分摊程序美术策划。本来想 solo 的不过觉得可能团队产出作品会更稳定完成度更好一点，所以最后还是决定组队参赛。

专职策划意味着不用自己给自己提需求，可以更多专注在设计上；另一方面要有把需求讲明白的能力，以及做好良好的团队沟通，保证游戏的完成度。所以我最开始定下的目标就是做一个【完整】的游戏——哪怕游戏的规模再小，表现力再薄弱，也是一个完整的有头有尾的游戏，有一个完整的体验。

（不过如果真的以拿奖为目标的话表现力的分数比完成度高，但考虑到我们团队表现力的能力也许完成度确实是比较可行的方案…… 大概？）

**合作**

我们开始开发是从 7/28 开始的，结束与 ddl 最后一点 8/20。中间看似有三周的时间，对我们来说的合作时间会更短：程序在同时为学校的另一个大项目做制作人兼主程；美术要准备因为疫情耽误到八月的期末考试；我则刚从隔离的小房间逃出来准备秋招。现在看来我们实际上有在认真同步合作的时间应该限于开发的最后一周多一点的时间，前两周基本上都不在同步工作——各自有各自的事。到最后阶段的时候才拿出敏捷开发的精神每天产出一个可玩的版本，辛亏我们一直在以完成游戏为标准不断改需求，最后才勉强做出一个流程上完整的游戏。

最开始的一周进度还好。几次脑暴会议的内容都控制的还好，然而我有些担心我是不是说话有点太多了，导致有时候会议会超时一些。不过决定方向和制作技术 demo 都还算顺利，技术 demo 到 8/7 也做出来了，美术也在积极找合适的画风和贴图。

但是团队在 8/6 到 8/12 之间的进度不顺利：这个阶段程序刚刚做好复制房间和旋转，但仍然有很多 bug，之前预计的钥匙 + 钥匙台 / 复制台并没有时间做；美术在积极寻找适合房间内装饰的画风，但是使用编辑器的时候也遇上一些问题；罪魁祸首的我这个时候正在经历搬家，之后又在折腾新家的网络，直到 11 号的时候才恢复沟通。这次团队内的合作脱节导致了后期进度的修改，并且在这个时候才开始修改钥匙 + 钥匙台的设计，但是程序觉得可能还是来不及做。我不得不因为自己失职而砍掉这个设计，并且因为之后没有时间做复制台，我不得不舍弃了之前设计的很多钥匙 + 钥匙台的关卡，虽然它很容易能改成复制台的设计。另一方面，由于剧情还没有完全确定下来，美术对于房间内装饰的风格也比较迷茫，各种元素也无从下手。这种情况直到我们恢复交流后才得以缓解。

基本上绝大多数的游戏内容都是最后一周做出来的。在恢复了团队沟通后，我们重新调整了工作方向，以每天做一个可玩的游戏版本为目标去完善当前的设计。程序因为还在做其他项目所以只能特定时间内完成功能实现，美术也考虑到工作量和解密玩法的问题而精简了房间的装饰。我意识到之前三个人开会可能效率会降低：讨论剧情的时候程序并不感冒，而讨论实现的时候美术能衔接的东西又很少。因此最后阶段主要由我当润滑油，每天分别和程序与美术沟通，所有的正式需求用共享文档的方式交流。之后的交流效率就高了很多。

我们是 ddl 前一天才完成游戏全部内容的，最后一天主要用来做 ppt 和视频。还是非常非常感激的我的两位队友，光耀和飞哥，谢谢你们没有在后期责怪我的交流问题，大家一起齐心协力把游戏完整的做出来了。真的非常非常感谢你们 w。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/8b0a5a0b-d5fd-40e7-a762-4843469d1932.png)
](https://davinci-cdn.lilithgame.com/forum/fd2ec789-6185-4ffb-9778-1a0c7c862453.png)

事实上，我们最开始的开发本来是按着计划走的，我也把需要做的目标和我自己关卡设计用的表格分享给队友们了，并且为了追踪进度专门做了一个流程迭代表格。

但是我没有好好提醒他们去及时更新共享文档的内容，这样我们就无法追踪各自的进度了。或者说——我觉得这个更可能——三个人的小团队其实不需要一个过于复杂的迭代表（这个表格有十几项要追踪的内容），更需要明确的是每个人分担的任务和 ddl，因为本来每个人负责的内容都是很模块化的。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/4d4f8c2e-31aa-4b90-af9f-c3fddef5289d.png)
](https://davinci-cdn.lilithgame.com/forum/e2769794-448b-4c43-aa38-e9354fb848fe.png)

因此后期的时候我们直接弃用了这个表格，转而用一个更直观的甘特图去表示任务。并且我每天都会和程序与美术交流确认今天要做的事（虽然有时候今天没空做），希望通过更简单的方式确定合作的内容。事实证明这种多次数的短时长交流 + 大目标在小团队（或者，迷你团队）会更有效一些。也许迭代表格这种东西适合大团队追踪整个项目的进度，但或者简化一些后也有一些参考价值。  
虽然这么说，我还是觉得关卡迭代表格是挺不可少的，可能因为今次是我自己一个人设计关卡，所以在程序和美术的队友眼中显得非常累赘——也许他们其实并不是很关心关卡的设计师傅符合心流或者教学模式，更关心自己的部分是否能完成。在这种小规模各司其事的工作氛围下，也许确实做好自己的部分，减少团队交流的时长而增加交流频率（短而多的交流）是一个更有效的解决方案吧。

[![](https://github.com/OkamiWong/clipped-web-pages/blob/master/Images/2020-9-9%2023-25-48/9973d341-bd47-4913-b301-e9fb12a55818.png)
](https://davinci-cdn.lilithgame.com/forum/72d9057c-fec9-468d-b7af-16a797973b96.png)

作为策划，很重要的工作就是制作管理：监督队友的工作任务与工作量，统计当前完成的任务并按需要调整方向，督促成品按时产出其实是很重要很重要的，哪怕是再小的团队甚至是 solo 也要注意各个 ddl 的安排，不然后期真的很吃瘪…… 不过也学到了很多东西。再次谢谢队友！！！！QAQ

**小结**

不知不觉好像写的有点长了…… 初衷还是和大家一起交流制作游戏的各种事情，如果你坚持看到这里也非常谢谢你啦~ 写了这么多还是试图展示我自己的一个制作流程，但其实我自己一点信心都没有，所以如果你对我上面写的内容有一些其他的看法或者建议意见请务必在下面留言告诉我，我也是游戏制作的一个小菜鸡，希望能和大家一起进步. jpg
